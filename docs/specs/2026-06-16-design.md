# AgentPipe 设计文档

工作名 AgentPipe(可改)。日期 2026-06-16。状态:待评审。

## 1. 背景与目标

### 1.1 要解决的问题

一个研发任务从想法到合入,作者已手动跑通一条固定流程:brainstorm 设计 → Claude 自查设计 → Codex 审设计 → 出执行文档 → 写码建 MR → code-review + simplify → Codex 审 MR → 按反馈改 → 再审,循环到干净。

流程本身已验证有效,缺的是把"人为手动串联"变成"可编排的自动化"。痛点有三:

- 步骤之间靠人记忆和手工切换工具,断点多、易漏。
- Codex 审查的"改到干净"循环要反复手动复制反馈、手动判断是否收敛。
- 不同任务需要的步骤不一样(有的不需要 Codex 审设计),但目前流程是脑子里的隐性约定,无法按任务增减。

### 1.2 目标

做一个个人用的本地编排客户端:

- 在 GUI 里为一个任务编排流程(增减步骤、调顺序、设参数),产出一份配置文件。
- 引擎解析配置并执行,把 Claude / Codex 当本地 CLI 黑盒调用。
- 执行过程支持人在回路:到关键点暂停、给出建议操作,由人批准 / 跳过 / 改参数 / 打断。
- 作者可持续迭代这套流程,工具本身轻量、改动成本低。

### 1.3 非目标

- 不并入其他上层应用,独立仓库独立演进(摸索期)。
- 不做分发(签名 / 公证 / 自动更新),只在本机 dev 模式跑。
- 不做团队协作 / 多用户 / 云端。
- 不替代 Claude Code 的交互式设计能力:brainstorm / writing-plans 这类需要人来回的步骤,仍在作者自己的 Claude Code 会话里做,本工具只负责编排与衔接。

## 2. 核心概念

| 概念 | 含义 |
|---|---|
| Task | 一次研发任务,对应一份 manifest 配置文件 |
| Manifest | 声明式配置,定义一个 Task 跑哪些 Step、顺序、循环、参数 |
| Step | 流程里的一个动作,有类型(claude / codex / human / loop) |
| Engine | 解析 manifest 并执行的无界面核心(Rust) |
| Run | 一次具体执行,持有运行时状态(每个 Step 的进度 / 状态) |
| Gate | 执行中的人工介入点:批准 / 跳过 / 改参数 / 打断 |
| Artifact | Step 产出的可被后续 Step 引用的东西(spec 路径 / MR URL / Codex 结论) |

## 3. 整体架构

### 3.1 三层逻辑分层

```
GUI(编排 + 控制台)
  │  启动时:写出 task.yaml,触发 run
  │  运行时:双向消息(引擎推进度 / GUI 发控制指令)
  ▼
配置(task.yaml)          ← 编排意图的快照,一次性交付
  ▼
引擎(headless 核心)      ← 只认两样:启动配置 + 运行时控制消息
  │  spawn 本地 CLI(黑盒)
  ▼
claude CLI / codex CLI
```

核心不变式:引擎完全不知道 GUI 存在。它的输入只有两样,启动时的 manifest 和运行时的控制消息;输出只有事件流。GUI 换成 CLI、换成将来并入其他上层应用,引擎一行不改。

### 3.2 Tauri 映射

Tauri 把"引擎"和"GUI"装进一个原生应用,但代码上严格分离:

- 引擎做成纯 Rust 库 crate(`engine`),不依赖 Tauri,可被独立的 CLI 二进制驱动(便于无界面测试和将来移植)。
- Tauri 后端(`src-tauri`)是薄绑定层:接收 webview 的 `invoke` 调用引擎,把引擎事件 `emit` 给 webview,负责 manifest 文件读写。
- 前端(`ui/`)是 React + Vite,跑在系统 webview 里,只做编排界面和运行控制台。

这样红线 1(核心与壳分离)在 Rust 内部就成立:`engine` crate 是可移植核心,`src-tauri` 是当前这层壳。

### 3.3 进程与子进程

```
Tauri App
 ├─ webview(React UI)
 └─ Rust 后端
     └─ engine crate
         ├─ Executor(状态机)
         ├─ ClaudeRunner ─ spawn ─▶ claude -p ...
         └─ CodexRunner  ─ spawn ─▶ codex exec ...
```

所有 CLI 在 Task 指定的 target 仓库目录(cwd)下执行。

## 4. 配置层:Task Manifest

### 4.1 Step 类型

四类。skill 不单列,作为 claude step 的一个字段(claude step 可引用一个 skill / slash 命令)。full-pipeline 走全自动路径,brainstorm / writing-plans 也由 claude step 自主跑(prompt 里约定"设计决策默认采用推荐选项,无需等人确认"),human 类型保留给确需人工产物交接的场景。

| kind | 语义 | 执行方式 |
|---|---|---|
| claude | 让 Claude 一次性完成一个指令(可引用 skill),一律最高权限 | spawn `claude --permission-mode bypassPermissions -p`,黑盒收结果 |
| codex | 让 Codex 审文档 / 审 MR / 问一句 | spawn `codex exec`,黑盒收结果 |
| human | 人去做(通常在自己的 Claude Code 会话),引擎等产物 | 引擎暂停,人提供 artifact 后批准继续 |
| loop | 包一段子步骤,满足条件或到上限才退出 | 反复执行 body,每轮判定 until |

### 4.2 manifest 示例

```yaml
version: 1
name: "实现 X 功能"
target: /abs/path/to/repo   # 所有 CLI 在此 cwd 下跑
mode: auto      # step = 每步前暂停等批准;auto = 自动跑,只在 human / gate 停

steps:
  - id: brainstorm            # 1. 出设计方案,决策默认采用推荐
    kind: claude
    skill: brainstorming
    prompt: "对这个任务做 brainstorming 产出设计方案,所有设计决策默认采用你的推荐选项无需等我确认,把方案写成设计文档落到 docs/specs/,最后只输出该文档路径"

  - id: design-review-claude  # 2. Claude 审设计
    kind: claude
    skill: four-dimension-review
    prompt: "审查设计文档 {{brainstorm.artifact}}"

  - id: design-review-codex   # 3. Codex 审设计;不需要时整段删掉即可
    kind: codex
    action: review-doc
    path: "{{brainstorm.artifact}}"

  - id: plan                  # 4. 出执行文档
    kind: claude
    skill: writing-plans
    prompt: "依据设计文档 {{brainstorm.artifact}} 产出执行文档(分步实施计划),落到 docs/plans/,最后只输出该文档路径"

  - id: implement             # 5. 写码 + 提交 + 建 MR
    kind: claude
    prompt: "按执行文档 {{plan.artifact}} 实现,提交代码,创建 MR,最后只输出 MR URL"

  - id: self-review           # 6. code-review + simplify 后提交
    kind: claude
    prompt: "对刚才的改动先跑 /code-review 修复发现的问题,再跑 /simplify 优化,然后提交到 MR"

  - id: codex-loop            # 7+8. Codex 审 MR → 改 → 再审,循环到干净
    kind: loop
    until: codex-clean
    max: 5                      # 防死循环上限
    body:
      - id: codex-review-mr
        kind: codex
        action: review-mr
        base: dev
      - id: apply-feedback
        kind: claude
        prompt: "按以下 Codex 反馈修改并提交:{{codex-review-mr.findings}}"

  - id: memorize              # 9. 总结记录点存入 / 更新记忆
    kind: claude
    prompt: "总结本轮任务需要被记录的点(踩坑/约定/关键设计决策),按记忆规范存入记忆并更新记忆索引"
```

增减流程 = 改这份 list。这就是"可编排"的全部交互面。

### 4.3 Step 间引用

引擎维护一个运行时上下文,保存每个已完成 Step 的产物。manifest 用 `{{step-id.field}}` 引用,执行前由引擎插值。可引用字段:

- `{{id.artifact}}` — human / claude step 产出的路径或字符串
- `{{id.findings}}` — codex step 的审查发现文本
- `{{id.verdict}}` — codex step 的结论(clean / changes_requested)

### 4.4 字段约定

claude step:`prompt`(必填,支持插值)、`skill`(可选,引用 skill 名)。无 timeout / 权限旋钮:claude 一律以 CLI 最高权限(bypassPermissions)跑(见 11.2),挂死靠控制台 Interrupt 兜底,不暴露 per-step 超时。

codex step:`action` ∈ {review-doc, review-mr, ask};review-doc 需 `path`;review-mr 需 `base`(目标分支);ask 需 `prompt`。

human step:`instruction`(给人看的说明)、`expects`(可选,artifact 表示需提供产物)。

loop step:`until`(退出谓词,Phase 1 仅支持 codex-clean)、`max`(最大轮数)、`body`(子步骤列表)。

## 5. 引擎执行模型

### 5.1 状态机

每个 Step 的状态:

```
Pending ─▶ Running ─▶ Done
   │          │
   │          ├─▶ AwaitingGate(human / step 模式的暂停)─▶ Running / Skipped
   │          └─▶ Failed
   └─▶ Skipped(被跳过)
```

Run 整体:Idle → Running → (Paused) → Finished(Success / Failed / Aborted)。

### 5.2 执行循环(伪代码)

```
fn run(manifest):
    ctx = Context::new(manifest.target)
    for step in flatten(manifest.steps):
        if manifest.mode == step:
            emit AwaitingGate(step, suggestion = describe(step))
            wait for ApproveGate | SkipStep | Abort
        result = execute(step, ctx)        # 见 5.3
        ctx.record(step.id, result.artifacts)
        if result.failed:
            emit StepFailed; pause for human decision
    emit RunFinished

fn execute(step, ctx):
    match step.kind:
        claude => ClaudeRunner.run(interpolate(step.prompt, ctx), step, ctx.cwd)
        codex  => CodexRunner.run(step.action, ..., ctx.cwd)
        human  => emit AwaitingGate(expects artifact); wait for Approve(artifact)
        loop   => loop_execute(step, ctx)   # 见 5.4
```

### 5.3 黑盒 CLI 调用

claude / codex 都是 spawn 子进程、流式读 stdout 行(转发给 GUI 当进度)、读 exit code、收尾解析。Step 内部不做实时可视化 Claude 的工具调用,只显示输出行 + 最终产物(黑盒约定)。

### 5.4 loop 执行

```
fn loop_execute(loop_step, ctx):
    for n in 1..=loop_step.max:
        emit LoopIteration(loop_step.id, n)
        for sub in loop_step.body:
            execute(sub, ctx)
        if eval_until(loop_step.until, ctx):     # codex-clean 见第 8 节
            emit LoopConverged(n); return Done
    emit LoopMaxReached(loop_step.max); pause for human decision
```

退出条件失败到达上限时不静默通过,暂停交人决定(继续加轮 / 接受现状 / 中止)。

## 6. 运行时控制协议

引擎暴露一对消息通道,Tauri 层桥接到 webview。引擎 crate 本身传输无关(用 Rust channel),不绑定 Tauri。

### 6.1 引擎 → GUI 事件

| 事件 | 载荷 |
|---|---|
| RunStarted | run_id, manifest 摘要 |
| StepStarted | step_id, kind |
| StepProgress | step_id, 文本行(CLI stdout 增量) |
| StepAwaitingGate | step_id, 建议操作描述, 是否需要 artifact |
| StepFinished | step_id, status, 产物摘要 |
| StepFailed | step_id, 错误摘要 |
| LoopIteration | loop_id, 第几轮 |
| LoopConverged / LoopMaxReached | loop_id, 轮数 |
| RunFinished | status |

### 6.2 GUI → 引擎指令

| 指令 | 载荷 |
|---|---|
| StartRun | manifest 路径 |
| ApproveGate | step_id, 可选 artifact |
| SkipStep | step_id |
| EditStepParams | step_id, 参数补丁(下一次执行生效) |
| Interrupt | 杀掉当前 step 子进程,Run 转 Paused |
| Resume | 从暂停处继续 |
| Abort | 结束整个 Run |

## 7. CLI Runner 封装

### 7.1 ClaudeRunner

调 `claude --permission-mode bypassPermissions -p <prompt>`,在 target cwd 下。要点:

- 流式读 stdout 转 StepProgress;stderr 收集进错误摘要。
- 一律带 `--permission-mode bypassPermissions`,让 headless claude 可自主 edit + bash(写码 / 提交 / 建 MR);不设 per-step 超时,挂死靠控制台 Interrupt 兜底。
- skill 引用:`skill: X` 时把 prompt 包成触发该 skill 的指令(例如前缀 `/X` 或在 prompt 里点名)。

### 7.2 CodexRunner

按 action 组命令,均带 `-s read-only`(Codex 只审不改)。注意 `codex exec review` 子命令专审仓库 git 改动,只用于 review-mr;审一份设计文档用通用 `codex exec`:

- review-doc(审设计文档,非 git 改动):`codex exec -s read-only --output-schema <schema> -o <out> -`,审查指令 + 文档内容走 stdin。
- review-mr(审仓库改动):`codex exec review --base <base> --output-schema <schema> -o <out>`。
- ask:`codex exec -s read-only -o <out> "<prompt>"`。

读 `-o` 输出文件,解析为结构化结果(见第 8 节)。review-doc 与 review-mr 都用第 8 节同一套 verdict / findings schema,故 loop 的 codex-clean 对两者通用。

## 8. Codex "改到干净" 契约

loop 的 `until: codex-clean` 需要机器可判定的收敛信号。约定 Codex review 走 `--output-schema` 强制结构化输出:

```json
{
  "verdict": "clean | changes_requested",
  "findings": [
    { "severity": "high|medium|low", "file": "...", "line": 0, "summary": "..." }
  ]
}
```

引擎解析 `-o` 文件:

- `verdict == "clean"` → loop 收敛退出。
- `verdict == "changes_requested"` → 把 findings 注入下一步 apply-feedback 的 prompt,继续循环。
- 解析失败 / 文件缺失 / verdict 缺失 → fail-closed,按 changes_requested 处理,绝不静默判为干净。

## 9. GUI

两块界面,React + Vite:

### 9.1 编排页(Composer)

- 可视化构建 / 编辑 task.yaml:增删步骤、拖拽排序、选 kind、填参数。
- 提供内置模板(full-pipeline 完整流程,见 4.2),复制后增减。
- 选 target 仓库目录、mode(step / auto)。
- 保存即写出 task.yaml。

### 9.2 运行控制台(Run Console)

- 步骤树:每个 Step 实时状态(Pending / Running / Awaiting / Done / Failed / Skipped),loop 显示轮次。
- 当前 Step 的 stdout 流式输出。
- 到 Gate 时弹出建议操作 + 按钮:批准 / 跳过 / 改参数 / 打断。
- human step 提供 artifact 输入框(路径或字符串)。
- Codex 审查结果以 findings 列表展示,loop 每轮可见收敛与否。

## 11. 错误处理与安全边界

### 11.1 错误路径

- CLI 不存在 / 非零退出 / 超时:Step 转 Failed,暂停交人决定,不静默继续。
- Codex 输出不可解析:fail-closed 按 changes_requested(见第 8 节)。
- loop 到上限未收敛:暂停交人决定,不静默通过。
- Interrupt:kill 当前子进程,Run 转 Paused,可 Resume / Abort。
- manifest 校验失败:启动前报错,带字段级 message,不进入执行。

### 11.2 写盘姿态

claude step 是真正执行外部代码的操作。本工具是个人本机用、target 由作者明确指定为可信仓库,故采用"最高权限 + 信任边界前置"姿态,而非 per-step 授权:

- claude step 一律带 `--permission-mode bypassPermissions`,可自主 edit + bash(写码 / 提交 / 建 MR)。不再有 `allow_writes` 旋钮——它曾用于区分只读/可写,但全自动 pipeline 每个 claude 步骤都要写盘,旋钮沦为噪声。
- 信任边界落在"选 target 仓库"这一步:GUI 在 auto 模式下显式告警"claude 将以 bypassPermissions 自主写码,确认 target 可信"。
- target 是作者明确指定的真实项目仓库,不做额外沙箱;但 cwd 严格取自 manifest.target,不回退到 home 目录。

### 11.3 边界

- 纯本地,无任何网络上报。
- 不持久化凭证;Claude / Codex 的登录态由各自 CLI 自管。

## 12. 非目标(YAGNI,Phase 1 不做)

- 条件分支(如"Codex 没意见就跳过 simplify")。先只做线性 + loop + gate。
- 并行步骤(claude 审与 codex 审同时跑)。先全串行。
- 多 Task 并发 / 无人值守过夜跑队列。
- 分发 / 签名 / 自动更新。
- until 谓词只实现 codex-clean,不做通用表达式引擎。

## 13. Phase 1 范围与里程碑

1. engine crate:manifest 解析 + 校验、Executor 串行执行、ClaudeRunner / CodexRunner、loop + codex-clean、Step 间插值。
2. cli crate:`agentpipe run task.yaml` 无界面跑通完整流程(先于 GUI 验证引擎)。
3. Codex 结构化审查契约打通(--output-schema → 解析 verdict)。
4. Tauri 壳 + 运行时协议桥接。
5. UI:编排页 + 运行控制台。
6. 内置 full-pipeline 模板(9 步全自动路径)。

建议 1→2→3 先把引擎在命令行跑通(可端到端 smoke),再上 4→5→6 的壳。
